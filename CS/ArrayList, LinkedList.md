# 데이터구조(배열과 연결리스트)

## 배열과 연결리스트란?

- 배열 : 같은 크기의 상자를 `빈틈없이 배열`
- 연결리스트 : 노드(상자)들을 `참조(위치연결)`하여 선으로 연결한 것
- 특정 상자를 찾으려면 배열은 인덱스를 이용해 한번에 찾고, 연결리스트는 끝에서 차례대로 확인해야하는 번거로움이 있다.
    

## 배열과 연결리스트의 장단점

- `탐색 : 배열 > 연결리스트`
- `데이터 추가, 삭제 : 배열 < 연결리스트`
- 배열은 리사이징이 필요하다.
    - 리사이징이란? : 공간이 부족한 경우 새로운 배열에 복사하는 것.
- 연결리스트 단점 : `추가적인 메모리를 할당`해야 한다.
    - 연결리스트의 노드를 구현할 때, `next`나 `prep`같은 참조자도 지정해야 하기 때문에 추가적인 공간이 더 필요하다.

## 어디에 사용되나?

- `해시 테이블` 구현,
    - 해시 테이블이란? : `배열의 장점(빠른 탐색) + 연결리스트의 장점(빠른 데이터 추가 삭제)`을 조합한 데이터 구조
    

# 탐색 알고리즘(해시/트리 등)

## 탐색 알고리즘이란?

- 데이터 정리 방법 : 자료구조, 데이터 찾는 방법 : 탐색 알고리즘
    
    *ex) 사전의 알파벳 순, 목차 등*
    
- 필요한 때에 필요한 데이터를 빠르게 찾기 위해서 데이터를 정리할 필요가 있다.
- 데이터를 찾을 때의 `데이터 구조와 데이터 저장 방식`에 따라 적합한 데이터 정리 방법이 필요.
- 처리 순서에 맞추어 데이터 구조를 정리할 필요가 있다.

## 어디에 사용되나?

### DBMS

- `인덱스가 없는 경우`
    - 원하는 데이터를 찾기 위해 처음부터 찾아야 한다. `(full scan)`
- `인덱스가 있는 경우`
    - 장점 : `탐색`에 용이
    - 단점 : 만약 데이터를 추가 및 삭제할 경우, 인덱스도 같이 갱신해야 한다.
        
        인덱스 갱신 때문에 `불필요한 오버헤드`가 발생할 수 있다.
        
        *ex) 서류철에 인덱스를 붙여놨지만 서류 추가시 인덱스를 수정해야하는 번거로움.*
        
    - `B 트리 인덱스`
        - `루트 블록 → 브렌치 블록 → 리프 블록 → 데이터 블록` 순으로 데이터를 찾는다. ⇒`1데이터 4블록`
        - 단점 :  `모든 데이터를 필요로 할 경우`, 테이블의 모든 블록과 인덱스 블록을 모두 읽어야 하기에 디스크 I/O가 증가한다. 인덱스 블록을 읽으면서 테이블 블록을 하나씩 읽기 때문에 읽어야할 블록수가 늘어나 풀스캔에 비해 속도가 느려진다.
            - **DBMS는 1회 디스크 I/O로 가능한 큰 크기의 데이터를 읽어서 I/O횟수를 줄이려한다.**

*ex) 책 읽는 방식*

- 풀 스캔 : 색인을 보지 않고 처음부터 읽음
- 인덱스 스캔 : 색인을 보면서 처음부터 읽음. 풀스캔에 비해 색인을 보면서 하는 시간이 추가됨.

### 해시 테이블

- `등호 검색`에 큰 강점을 보임.
- `키와 값 조합`으로 표를 구성한 데이터 구성
- 키는 **해시 함수**로 인해 `해시값으로 변환`되며, **해시값은 고정 길이**이기 때문에 검색에 아주 용이하다.
- 데이터 양이 늘어나도 **등호 검색의 속도는 변하지 않는다.**
- **범위 탐색에는 적합하지 않다.**
