# Big-O, 오메가 표기법

- 최악의 경우 (Big-O), 최선의 경우(오메가)
- 알고리즘의 효율을 따질 때는 오메가보다는 Big-O가 더 중요하다.
- 예시
    - Big-O : O(n)
    - 오메가 : 오메가(n) → 시계 브랜드 로고를 보면 알 수 있음.

## 값 순서

(높은것)n^2 → nlongn → n → logn → 1(낮은것)

# 알고리즘 종류별 Big-O, 오메가

## 선형탐색과 이진탐색

### 선형탐색, 이진탐색이란?

- 선형탐색(linear search) : 모든 n개의 자료를 하나씩 차례대로 확인하는 작업
- 이진탐색(binary search) : 모든 n개의 자료 중, 가운데 자료를 기준으로 판단하여 추리는 작업

## 버블정렬과 선택정렬

### 버블정렬, 선택정렬이란?

- 버블정렬(bubble sort)
    - 구름위르 떠다니는듯이 두개의 값을 비교 후 정렬하는 방법
    - (n-1) * (n-1) = n^2 + 2n + 1
- 선택정렬(selection sort)
    - 모든 정수 중 가장 작은 값을 찾아 해당하는 위치와 변경시키는 방법
    - (n-1)번 → (n-2)번 → … → 1번 = n(n+1)/2

## 종류별 Big-O, 오메가 정리

| 값 | Big-O | 오메가 |
| --- | --- | --- |
| n^2 | 버블정렬, 선택정렬 | 버블정렬, 선택정렬 |
| nlongn |  |  |
| n | 선형탐색 |  |
| longn | 이진탐색 |  |
| 1 |  | 선형탐색, 이진탐색 |

## 정수형과 실수형

정수형에는 byte, short, int, long, 실수형에는 float, double이 있다.

**Var8**

```java
public class Var8 {
    public static void main(String[] args) {

        byte b = 127; //-128 ~ 127
        short s = 32767; // -32,768 ~ 32,767
        int i = 2147483647; //-2,147,483,648 ~ 2,147,483,647 (약 20억)
        long l = 9223372036854775807L; //-9,223,372,036,854,775,808 ~ 9,223,372,036,854,775,807
        
        //실수
        float f = 10.0f; 
        double d = 10.0;
    }

```

- 각 값의 범위는 주석과 같다.


## 프리미티브 타입 종류

`자주 사용하는  타입`과 `자주 사용하지 않는 타입`으로 나누어 정리해본다.

### 자주 사용하는 타입

- `정수 - int, long` : 기본으로 int를 사용하나, 값이 20억이 넘어간다면 long을 사용하자.
- `실수 - double` : float이 있으나 자주 사용하지 않는다. double을 사용하자. 그 이유는 아래에 있다.
- `불린형 - boolean` : true와 false중 하나의 값을 갖는다. 보통 조건문의 참 거짓에 사용한다.
- `문자열 - String` : 문자가 하나일 경우, ‘char’가 있지만, 문자열로 통용해서 사용하도록 한다.

> float, char를 사용하지 않는 이유 : 최근 메모리는 매우 저렴해졌다. 해당 변수를 사용하여 메모리를 줄이기 보다는 개발 속도나 효율에 초점을 맞추는 것이 더 효과적이다.
> 

## 자주 사용하지 않는 타입

- `정수형 - `byte`` : 표현 길이가 너무 작다. 또 자바는 기본으로 4byte( `int` )를 효율적으로 계산하도록 설계되어 있다. `int`를 사용하자.
    - `byte` 타입을 직접 선언하고 여기에 숫자 값을 대입해서 계산하는 일은 거의 없다.
    - 대신에 파일을 바이트 단위로 다루기 때문에 파일 전송, 파일 복사 등에 주로 사용된다.
- `정수형 - `short`` : 표현 길이가 너무 작다. 또 자바는 기본으로 4byte( `int` )를 효율적으로 계산하도록 설계되어 있다. `int` 를 사용하자
- `실수형 - `float`` : 표현 길이와 정밀도가 낮다. 실수형은 `double` 을 사용하자.
- `문자형 - `char`` : 문자 하나를 표현하는 일은 거의 없다. 문자 하나를 표현할 때도 문자열을 사용할 수 있다.
    - 예를 들어 `String a = "b"` 와 같이 사용하면 된다.

# `리터럴`

- 리터럴 : 개발자가 직접 적은 고정된 값을 나타내는 프로그래밍 용어

## 리터럴 타입 지정

- **정수형은 int를 기본으로 사용한다.** int 범위인 약 20억을 넘어가면 long으로 대체하여 사용하도록 하자. long은 정수형에 ‘l’, ‘L’을 붙여 사용하는데 ‘l’은 숫자 1, 알파벳 l과 구분하기 힘드므로 대문자 ‘L’을 사용하도록 하자.
- **실수형은 double을 기본으로 사용한다.** float은 f를 붙여서 사용한다.

# `변수 선언 및 초기화하는 방법`

- 변수가 필요한 이유
    - 사용자의 입력이나 코드 내 여러 부분의 값을 한번에 바꾸고 싶다면 그 값을 담은 그릇이 필요하다.
    - 그 그릇의 역할이 바로 변수이다.

**Var1**

```java
public class Var1 {
    public static void main(String[] args) {
        int a; // 변수 선언 
        a=10; // 변수 초기화
        
        System.out.println(a);
        System.out.println(a);
        System.out.println(a);
    }
}
```

- `int a`
    - int : 정수 타입의 값을 넣는 그릇
    - a : 그릇 이름
- `a = 10;`
    - a라는 변수에 10이라는 값을 넣는다.
    - a : 그릇 이름
    - = : 수학에서는 “같다”라는 의미라면, 개발에서는 a 변수에 값을 `대입` 하라는 의미이다.

## 변수 선언

변수를 선언하면 컴퓨터 안의 **메모리 공간을 확보**하여 그곳에 데이터를 저장할 수 있다. 변수라는 이름을 통해 메모리 공간에 접근할 수 있는 것이다.

**Var4**

```java
public class Var4 {
    public static void main(String[] args) {
        int a; // 변수 선언
        int b; // 변수 선언
        int c,d; // 변수 한번에 여러개 선언
    }
}
```

- 변수 선언은 같은 자료형일 경우, 한번에 여러개 선언이 가능하다.

### 변수 선언 규칙

규칙과 관례로 나눈다.

규칙

- 변수 이름은 숫자로 시작할 수 없다.(예: 1st)
- 변수 이름은 숫자로 시작할 수 없다. (예: `1num` , `1st` )
    - 그러나 숫자를 이름에 포함하는 것은 가능하다 (예: `myVar1` , `num1` ).
- 이름에는 공백이 들어갈 수 없다.
- 자바의 예약어를 변수 이름으로 사용할 수 없다. (예: `int` , `class` , `public` )
- 변수 이름에는 영문자(`a-z` , `A-Z` ), 숫자(`0-9` ), 달러 기호(`$` ) 또는 밑줄(`_` )만 사용할 수 있다.

**관례**

> 클래스는 대문자로 시작, 나머지는 소문자로 시작 + 카멜표기법
> 
- 자바에서 클래스 이름의 첫 글자는 대문자로 시작한다. 그리고 나머지는 모두 첫 글자를 소문자로 시작한다. 여기에 낙타 표기법을 적용하면 된다. 이렇게 하면 모든 자바 관례를 다 외울 수 있다!
- 예시: 클래스는 첫 글자 대문자, 나머지는 모두 첫 글자 소문자로 시작 + 낙타 표기법
    - 클래스: `Person` , `OrderDetail`
    - 변수를 포함한 나머지: `firstName` , `userAccount`
- 여기에 예외가 딱 2개 있다.
    - 상수는 모두 대문자를 사용하고 언더바로 구분한다. (상수는 뒤에서 학습)
        - `USER_LIMIT`
    - 패키지는 모두 소문자를 사용한다. (패키지는 뒤에서 학습)
        - `org.spring.boot`

## 변수 초기화

- 초기화 : 변수를 선언하고 선언한 변수에 처음으로 값을 저장하는 것

**Var5**

```java
public class Var5 {
    public static void main(String[] args) {
        int a; // 변수 선언과 초기화를 따로
        a = 10; // 변수 초기화
        System.out.println(a);

        int b = 10; // 변수 선언과 초기화를 동시에
        System.out.println(b);

        int c = 30, d = 40; // 여러 변수 선언과 초기화를 동시에
        System.out.println(c);
        System.out.println(d);
    }
}
```

### 변수 초기화는 반드시 해야한다.

메모리 공간은 자바 뿐만 아니라 다른 모든 시스템이 공유하는 공간이다. 변수를 선언하면 메모리 공간을 차지하게 되는데, 그 공간에 기존에 사용하던 어떤 값이 존재했는지 알 수 없다. 따라서 초기화를 하지 않으면 다른 값이 출력될 수 있다. 이런 문제를 해결하기 위해 자바에서는 강제로 진행시킨다.

> 지역 변수는 개발자가 직접 초기화 해주어야 하지만, 클래스 변수와 인스턴스 변수는 자바가 자동으로 초기화해준다. 추후 배우도록 한다.
> 

**Var6**

```java
public class Var6 {
    public static void main(String[] args) {
        int a;
        System.out.println(a); // 컴파일 에러 발생
    }
}
```

- 컴파일 에러 발생 : `java: variable a might not have been initialized`
    - 컴파일 에러 : 자바 컴파일러가 컴파일 할 때, 문법 오류 등과 같이 문제가 발생했을 경우, 프로그램을 강제로 중지시키는 에러

## 변수 값 변경

- a 라는 그릇에 기존에 10을 넣었다면 그 이후 50으로 변경하면 10이 제거되고 50으로 바뀐다.

**Var3**

```java
public class Var3 {
    public static void main(String[] args) {
        int a; // 변수 선언
        a=10; // 변수 초기화

        System.out.println(a); // 10 출력
        a = 50; // 변수 값 변경
        System.out.println(a); // 50 출력
    }
}
```



# 스프링

# IoC(Inversion of Control, 제어의 역전)

- 개발자가 아닌 다른 무언가에서 프로그램 흐름 관리를 대신 해주는 것
- 기존에는 클라이언트 객체(OrderServiceImpl)가 MemoryMemberRepository, FixDiscountPolicy 객체를 직접 생성하고 연결하고 실행함으로써 직접 흐름을 제어했다.
- 반면, AppConfig가 등장하면서, 클라이언트 객체는 자신의 역할만 잘 수행하면 되고, 나머지 설정 부분은 AppConfig에 일임했다. 즉, 클라이언트 객체는 어떤 객체가 연결될지 모른다.
- 프로그램에 대한 흐름 제어권은 AppConfig가 모두 갖고 있다.
- 이처럼 프로그램 흐름 제어를 직접하는 것이 아니라, 외부에서 관리하는 것을 제어의 역전(IoC)라고 한다.

## 프레임워크 vs 라이브러리

- 프레임워크 : 내가 작성한 코드를 제어하고, 대신 실행한다.
- 라이브러리 : 내가 작성한 코드가 프로그램의 흐름을 제어한다.

# DI(Dependency Injection, 의존 관계 주입)

- 클라이언트 객체는 추상(인터페이스)와만 연결되어있다. 해당 인터페이스를 통해 어떤 구현체가 연결될지는 모른다.
- 구현체 연결은 AppConfig가 담당한다.
- 이처럼 구현체를 클라이언트 객체 본인이 아닌 외부에서 주입시켜주는 것을 의존 관계 주입이라고 한다.
- 의존관계는 정적인 클래스 의존관계와 동적인 객체 인스턴스 의존관계로 나누어 생각한다.

### 정적인 클래스 의존관계

- Import를 통해 연결되는 클래스간 의존관계
- 클래스 다이어그램을 통해 확인할 수 있다.

### 동적인 객체 인스턴스 의존관계

- 애플리케이션 실행 시점에 외부에서 구현 객체를 생성하고 클라이언트 객체에 주입시켜주는 것
- 객체 다이어그램을 통해 확인할 수 있다.

### DI를 사용하는 이유

- 클라이언트 코드를 변경하지 않고, 클라이언트가 호출하는 객체(인스턴스)를 변경할 수 있다.
- 정적인 클래스를 변경하지 않고, 동적인 객체 인스턴스 의존관계를 쉽게 변경할 수 있다.
